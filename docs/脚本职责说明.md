# 脚本职责说明

## `watchdog.py`

这是后台常驻核心。

职责：

- 轮询任务状态
- 读取日志
- 推断阶段
- 判断 `slow / stalled / stopped / completed`
- 写 `status.json`
- 写 `events.log`

如果只能先看一个脚本，就先看它。

## `status.py`

这是终端展示层。

职责：

- 读取 `status.json`
- 用终端 HUD 风格输出状态
- 支持一次性查看和 `--watch` 持续刷新

它不负责监控逻辑，只负责展示。

## `start_watchdog.sh`

这是后台启动入口。

职责：

- 清理旧的运行态残留
- 后台启动 watchdog
- 告诉你状态文件和日志在哪里

## `stop_watchdog.sh`

这是停止入口。

职责：

- 结束当前 watchdog 进程
- 清理 pid 文件

## `restart.sh`

这是手动确认后的执行器。

职责：

- 从 `status.json` 里读取 `restart_command`
- 在你确认后真正执行它

它不是 watchdog 的唯一重启入口。
自动恢复时，watchdog 会直接调用底层的 `tmux_restart.sh`。

## 脚本之间的关系

```text
被监听任务
  ↓ 写日志
watchdog.py
  ↓ 写状态
status.json / events.log
  ↓ 读取
status.py

用户确认重启
  ↓
restart.sh
```

## 最容易混淆的点

### `status.py` 不是 watchdog

它只是看板，不是监控器。

### `restart.sh` 不是唯一的自动恢复链路

它只是给用户手动触发重启的入口。
真正的自动恢复链路在 `watchdog.py -> tmux_restart.sh`。
